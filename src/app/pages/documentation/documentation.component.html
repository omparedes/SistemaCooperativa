Documento de Arquitectura y Especificación Técnica

SistemaCooperativa (TailAdmin)

Fecha: 17 de Diciembre de 2025
Versión: 1.0
Framework: Angular (Standalone) + Tailwind CSS 4

1. Resumen Ejecutivo

El proyecto SistemaCooperativa es una aplicación web administrativa basada en el template TailAdmin, desarrollada sobre Angular (Versión experimental/reciente ^20.x) utilizando la arquitectura de Standalone Components. El sistema está diseñado para gestionar actividades de una cooperativa o mercado, incluyendo módulos para Socios, Gastos, Recaudación e Inventario.

Estado Actual: La aplicación opera como un prototipo de alta fidelidad. La interfaz de usuario (UI) está implementada y es funcional, pero la persistencia de datos se realiza mediante Mock Data (Hardcoded) o LocalStorage. No existe actualmente integración con un backend real.

2. Visión General de la Arquitectura

La aplicación sigue una arquitectura Monolítica Modular basada en Componentes (Standalone). Se ha eliminado la dependencia de NgModules para la mayoría de la aplicación.

Característica Implementación
Bootstrapping bootstrapApplication en main.ts
Ruteo Centralizado en app.routes.ts, mayormente Eager Loading
Estilos Tailwind CSS v4 (configuración en @theme)
Gestión de Estado Local (en componentes) y LocalStorage

4. Descripción de Módulos

Dado que la aplicación usa Standalone Components, el concepto de "Módulo" es casi inexistente.

Estado de Migración

Actualmente el proyecto es 100% Standalone. Se han eliminado todos los NgModules legacy (como SociosModule, GastosModule, etc.) para simplificar la arquitectura y reducir el boilerplate.

5. Catálogo de Componentes Clave

5.1. Feature: Socios

  • Componente: SocioListComponent
  • Responsabilidad: Listar socios con filtrado.
  • Datos: Array estático hardcoded.

5.2. Feature: Gastos

  • Componente: GastoListComponent
  • Responsabilidad: CRUD completo de gastos.
  • Datos: Persistencia en localStorage.

5.3. Layout (Shared)

  • Componente: AppLayoutComponent
  • Responsabilidad: Orquestador principal (Sidebar + Header).

6. Catálogo de Servicios

Los servicios globales se han centralizado en src/app/core/services. Actualmente gestionan principalmente el estado de la UI (Sidebar, Theme, Modales). Se planea agregar aquí los servicios de negocio (HTTP).

Servicio Responsabilidad
SidebarService Gestiona estado del menú lateral.
ThemeService Gestiona tema Claro/Oscuro.
ModalService Controla apertura de modales simples.

7. Arquitectura de Ruteo

  • Estrategia: Eager Loading (Carga ansiosa).
  • Ruta raíz /: Redirige a Dashboard.
  • Riesgo: Bundle inicial pesado. Se recomienda migrar a Lazy Loading (loadComponent).

8. Recomendaciones de Desarrollo

Fase 1: Arquitectura

  • Crear directorio Core para servicios singleton.
  • Implementar Lazy Loading en rutas.
  • Eliminar NgModules remanentes.

Fase 2: Integración Backend

  • Crear servicios HTTP (SociosService, GastosService).
  • Reemplazar Mocks por llamadas API.
  • Configurar variables de entorno.

9. Guía Funcional de Vistas

1. Métricas Dashboard

Path: / (EcommerceComponent)

Contenido: Vista principal que muestra tarjetas de resumen (KPIs) con totales de Ingresos, Gastos y Beneficio Neto del mes en curso.

Funcionamiento: Carga datos simulados para gráficos de línea y barras. Sirve como punto de entrada para visualizar el estado general de la cooperativa.

2. Usuario (Perfil)

Path: /profile (ProfileComponent)

Contenido: Información detallada del usuario logueado, incluyendo foto de perfil, rol y datos de contacto.

Funcionamiento: Permite visualizar tarjetas de información personal. Actualmente utiliza datos estáticos de ejemplo.

3. Listado de Socios

Path: /socios (SocioListComponent)

Contenido: Tabla con la lista completa de socios (aprox. 100 registros precargados). Muestra ID, Tienda, Nombre, Estado y Deuda Total.

Funcionamiento: Incluye un buscador en tiempo real por nombre, apellido o número de tienda. Permite navegar al detalle de cada socio.

4. Información del Socio

Path: /socios/:id (SocioDetailComponent)

Contenido: Vista detallada de un socio específico. Muestra su historial de pagos, deudas pendientes y estado de habilitación.

Funcionamiento: Recibe el ID por URL, busca al socio en el array estático y muestra sus datos. Incluye botón para "Registrar Pago".

5. Registrar Pago

Path: /pagos/registrar/:id (RegistroPagoComponent)

Contenido: Wizard de 3 pasos para procesar un pago: 1) Selección de Conceptos, 2) Confirmación de Monto, 3) Generación de Recibo.

Funcionamiento: Valida montos y conceptos. Al finalizar, actualiza (simuladamente) la deuda del socio y genera un registro de transacción.

6. Recaudación Diaria

Path: /pagos/recaudacion-diaria (RecaudacionDiariaComponent)

Contenido: Resumen de caja del día. Muestra el total recaudado en efectivo y transferencias, desglosado por cajero.

Funcionamiento: Simula el cierre de caja diario, permitiendo visualizar cuadres y diferencias (mock).

7, 8, 9. Reportes (Ingresos, Gastos, Almacén)

Path: /reportes/* (ReportesComponent)

Contenido: Vistas gráficas y tabulares para análisis histórico de datos.

Funcionamiento: Componente reutilizable que adapta su contenido según la ruta (`/reportes`, `/gastos`, `/inventario`). Usa gráficos de ApexCharts para visualizar tendencias.

10. Gastos

Path: /gastos (GastoListComponent)

Contenido: CRUD para la gestión de egresos operativos y administrativos.

Funcionamiento: Permite crear, editar y eliminar gastos. Persiste la información en LocalStorage para mantener los datos entre recargas.

11. Movimientos Bancarios

Path: /bancos (BancosComponent)

Contenido: Registro de depósitos, retiros y conciliaciones bancarias.

Funcionamiento: Interfaz para registrar manualmente operaciones bancarias externas asociadas a la cooperativa.

12. Inventario - Almacén Interno

Path: /inventario (InventarioListComponent)

Contenido: Control de stock de insumos internos (limpieza, papelería, etc.).

Funcionamiento: Control de kardex básico (entradas y salidas). Alertas visuales de stock bajo.

13. Auditoría de Cambios

Path: /auditoria (AuditoriaComponent)

Contenido: Log histórico de acciones críticas realizadas en el sistema (quién hizo qué y cuándo).

Funcionamiento: Lista de solo lectura que registra eventos de modificación de datos sensibles.
